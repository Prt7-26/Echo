01项目定位

Hermes Agent 是 Nous Research 出品的自进化 AI 代理。和普通"会调工具的 LLM 包装器"相比，它的核心差异是内置学习闭环：

  • 技能 (Skills)：从经验自动创建、在使用中改进
  • 记忆 (Memory)：主动持久化，跨会话搜索 (FTS5 + LLM 摘要)
  • 用户建模：通过 Honcho 辩证式建模
  • 多平台同人格：CLI、Telegram、Discord、Slack、WhatsApp、Signal、邮件等共享同一会话
  • 定时自动化：内置 cron 调度器，向任何平台投递
  • 子代理委派：生成隔离子代理处理并行工作流，零上下文开销
  • 研究就绪：批量轨迹生成可直接用于训练下一代工具调用模型

支持几乎所有主流模型——OpenAI、Anthropic、Nous Portal、OpenRouter (200+ 模型)、NVIDIA NIM、小米 MiMo、z.ai/GLM、Kimi/Moonshot、MiniMax、HuggingFace，以及任意自定义 OpenAI 兼容端点。hermes model 一键切换，无锁定。

03入口点（三条路径）

pyproject.toml 的 [project.scripts] 定义了三个 console scripts：

命令入口用途
hermes hermes_cli.main:main 交互式 CLI，默认子命令 chat，还有 model / setup / tools / gateway / cron / dashboard / doctor 等
hermes-agent run_agent:main 编程式调用 AIAgent，给脚本 / 测试 / 批跑用
hermes-acp acp_adapter.entry:main ACP 协议服务，供 IDE 插件 (VS Code / Zed / JetBrains) 用

hermes shell 脚本和 hermes_bootstrap.py 只做 Windows UTF-8 编码修复——所有入口点都先经过它。

04Agent 主循环

所有用户表面最终都汇聚到 run_agent.py 的 AIAgent.run_conversation()：

while api_call_count < max_iterations:
    response = client.chat.completions.create(messages, tools)
    if response.tool_calls:
        results = handle_function_call(...)   # 派发到 model_tools.py
        messages.append(tool_results)
    else:
        return response   # 最终输出

消息格式统一用 OpenAI 格式 ({"role": "system/user/assistant/tool"})，即使底层是 Anthropic、Gemini、Bedrock 也由 adapter 转换。关键支持文件：

agent/conversation_loop.py主循环外层包装 + system prompt 注入
agent/tool_executor.py工具调用执行
agent/tool_dispatch_helpers.py工具调用编排（并行、错误包装）
agent/prompt_builder.py消息装配 + prompt cache 控制
agent/context_engine.py上下文文件/引用注入
agent/memory_manager.py长期记忆管理
trajectory_compressor.pytoken 紧张时降级压缩

05Provider 抽象

核心策略：所有 provider 都以 OpenAI SDK 为统一客户端接口。原生 Anthropic、Bedrock、Gemini、Codex 由 adapter 转换成 OpenAI 格式：

  • agent/anthropic_adapter.py
  • agent/bedrock_adapter.py
  • agent/gemini_native_adapter.py  /  gemini_cloudcode_adapter.py
  • agent/codex_responses_adapter.py
  • agent/azure_identity_adapter.py

providers/base.py 是基础接口，plugins/model-providers/ 下是各家具体适配（OpenRouter、Vercel、Modal、Daytona、GMI 等）。

06工具 & 技能系统（核心扩展点）

这是二次开发最容易也最值得切入的地方。

工具 (Tools)

  • 位于 tools/，约 79 个
  • 通过 registry 模式自动发现：每个 tools/*.py 在导入时调用 registry.register(name, schema, handler, check_fn, requires_env)
  • model_tools.py 的 handle_function_call() 按名字派发
  • toolsets.py 定义命名集合（research、development、safe、full_stack 等）
  • toolset_distributions.py 定义"分布"——批量跑时按概率抽样工具组合
  • tools/lazy_deps.py：首次使用才装依赖，缩小供应链攻击面

代表性工具：file_tools、code_execution_tool、browser_tool、mcp_tool、delegate_tool（子代理委派）、memory_tool、cronjob_tools。

技能 (Skills)

  • 每个技能是一个文件夹，含 SKILL.md（元数据）+ Python 处理器 + 资源
  • 兼容 agentskills.io 开放标准
  • 内置技能 27 个：github、email、research、media、devops 等
  • 可选技能 17 个：金融、健康、区块链、red-teaming 等
  • 用户自建技能放在 ~/.hermes/skills/，可执行后自我改进
  • 技能命令以 user message 注入（不进 system prompt），保护 prompt cache

插件 (Plugins)

更深的扩展点，每个插件有 manifest.json，由 hermes_cli/plugins.py 加载。例子：memory（Honcho/Mem0/Supermemory）、model-providers、kanban（多代理协作）、observability、browser。

08状态持久化

hermes_state.py 是一个单体 SQLite 仓库（~/.hermes/state.db）：

  • 单类 SessionDB，WAL 模式（并发读 + 单写）
  • 写入用 BEGIN IMMEDIATE + 退避重试，避免高并发下的车队效应
  • FTS5 全文索引用于跨会话搜索
  • 存储：会话、消息、checkpoints、审批记录
⚠ 二次开发警告
表结构改动牵一发动全身。hermes_state.py 是最不该轻动的文件——任何 schema 变更必须写迁移 + 跑全套 tests。

10Runner 矩阵

run_agent.py

AIAgent 主类与编程入口，~60 个初始化参数。hermes-agent 命令直接调它，是脚本和测试的主要入口。

batch_runner.py · 你问的就是这个

它是 "把 Agent 跑成数据工厂" 的并行运行器，专门用于批量评测和生成训练数据。具体能力：

  • 多进程并行：用 multiprocessing.Pool 同时跑 N 个 prompt（--batch_size）
  • 数据集加载：读 JSONL 数据集（--dataset_file=data.jsonl）
  • 断点续跑：写 checkpoint，--resume 可从崩溃位置继续
  • 轨迹保存：以 sharegpt 风格的 from/value 对格式输出，可直接喂给 HuggingFace 训练流水线
  • 工具用量统计：跨所有 batch 聚合（每个工具的调用次数、成功率、失败率），schema 自动对齐避免 HF datasets 加载时报错
  • 工具集分布抽样：通过 --distribution=image_gen 等指定，每次抽取不同的工具组合，制造多样性训练样本（见 toolset_distributions.py）
  • 推理控制：--reasoning_disabled 或 --reasoning_effort=medium 控制 reasoning token 预算
  • prefill 消息：--prefill_messages_file 可以预填对话历史（蒸馏特定模型行为时用）

典型用例

# 跑 evals
python batch_runner.py --dataset_file=eval.jsonl --batch_size=10 --run_name=eval_v1

# 生成图像生成场景的训练数据
python batch_runner.py --dataset_file=prompts.jsonl --batch_size=20 \
                       --run_name=imggen --distribution=image_gen

# 崩溃后续跑
python batch_runner.py --dataset_file=prompts.jsonl --batch_size=20 \
                       --run_name=imggen --resume

# 用 Opus 的 prefill 模板蒸馏 Opus 行为到目标模型
python batch_runner.py --dataset_file=prompts.jsonl --batch_size=10 \
                       --run_name=distill --prefill_messages_file=configs/prefill_opus.json

它是 Hermes "研究就绪 (research-ready)" 这个卖点的核心实现——README 里说的 "用于训练下一代工具调用模型" 就靠它。

mini_swe_runner.py

软件工程任务的专用 runner（代码审查、debug、重构），用任务专属的 prompt 和工具约束包装 AIAgent。

11终端后端（六种执行环境）

tools/environments/ 提供统一接口的六种 sandbox：

后端类型典型用途
Local本机 shell开发、单机日常
Docker容器隔离的本地开发
SSH远程挂在 VPS 上跑
ModalServerless按需 wake，闲时几乎零成本
DaytonaServerless云端持久工作区
SingularityHPC 容器研究集群

这是 Hermes "$5 VPS 或 GPU 集群都能跑"的关键——前端 Agent 不变，只换执行后端。

12依赖策略（供应链安全敏感）

pyproject.toml 注释里写了非常详细的安全政策：

  • 核心依赖全部精确锁定 ==X.Y.Z，绝不使用范围
  • 起因：2026-05-12，mistralai 2.4.6 被 Mini Shai-Hulud 蠕虫攻击
  • 任何 provider 特定依赖（anthropic、firecrawl、exa、fal、edge-tts）必须放到 [project.optional-dependencies]，通过 tools/lazy_deps.py 懒加载
  • 新增依赖必须更新 uv.lock
⚠ 二次开发约束
你新增依赖时必须遵守此政策。若引入范围版本或把可选依赖塞进核心 dependencies，会扩大下游用户的供应链攻击面。

13二次开发切入点（按改动成本从低到高）

难度改什么怎么做
⭐ 加一个工具 在 tools/ 加 mytool.py，调用 registry.register(...)。零侵入。
⭐ 加一个技能 在 skills/ 或 ~/.hermes/skills/ 加文件夹 + SKILL.md。零侵入。
⭐⭐ 接新 LLM provider 大多数 OpenAI 兼容端点直接 hermes model 配置；原生协议在 plugins/model-providers/ 加插件。
⭐⭐ 加新消息平台 在 gateway/platforms/ 实现 Platform 子类，注册到 gateway/platform_registry.py。
⭐⭐ 加 CLI 子命令 在 hermes_cli/commands.py 的 COMMAND_REGISTRY 里加 CommandDef，处理函数放到 hermes_cli/ 的新文件。
⭐⭐⭐ 改 Agent 主循环行为 慎改 run_agent.py / conversation_loop.py。优先考虑 hook 机制（gateway/hooks.py、agent/shell_hooks.py）。
⭐⭐⭐⭐ 改状态层 hermes_state.py。高风险：写迁移 + 跑 tests/ 全套。

14测试基线

tests/ 有约 1,100 个测试文件。pytest 配置：

addopts = "-m 'not integration' -n auto --timeout=30 --timeout-method=signal"

开发循环：

./setup-hermes.sh             # 初始化 venv 和符号链接
python -m pytest tests/ -q    # 全套（含 timeout 防 hang）

pyproject.toml 注释解释了为什么必须用 --timeout：测试线程泄漏会在会话结束时死锁。
